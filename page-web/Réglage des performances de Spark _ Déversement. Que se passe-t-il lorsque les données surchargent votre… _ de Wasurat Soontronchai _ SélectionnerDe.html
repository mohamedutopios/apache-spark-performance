Que se passe-t-il lorsque les données surchargent votre mémoire ?

Un problème de déversement se produit lorsque le déplacement d'un RDD (ensemble de données distribués résilient, alias structure de données fondamentale dans Spark) passe de la RAM au disque, puis revient à nouveau à la RAM.

En termes simples, ce comportement se produit lorsqu'une partition de données donnée est trop grande pour tenir dans la RAM de l'exécuteur. Spark lira et écrira les données excédentaires sur le disque afin de libérer de l'espace mémoire dans la RAM locale pour les tâches restantes de la tâche.

C'est un processus coûteux et lent !

Dans cet article, nous passerons en revue tous les sujets ci-dessous pour créer une compréhension claire du problème de déversement dans Spark.

  • Structure de la mémoire dans Spark Cluster
  • Action possible pour générer un déversement
  • Surveillance des déversements
  • Atténuation des déversements

Structure de la mémoire dans Spark Cluster

Commençons par comprendre la structure de la mémoire dans Spark Worker Node. Disons que nous avons un cluster Spark avec un exécuteur dans le nœud de travail (par défaut Databricks Cluster ).

La structure de la mémoire à l'intérieur du nœud de travail ressemblerait à ceci.

Comme vous pouvez le constater, les zones de mémoire du nœud de travail sont la mémoire sur le tas , la mémoire hors tas et la mémoire supplémentaire.

Ce qui est réellement impliqué dans le problème de déversement, c'est la mémoire sur le tas.
 Cette mémoire est utilisée pour les tâches et le traitement lors de la soumission d'une tâche Spark. La zone Mémoire sur le tas comprend 4 sections.

  1. Mémoire réservée (~ 300 Mo) :
    Cette section est dans un premier temps réservée par Spark à des fins internes. Il est utilisé pour garantir la suffisance de la mémoire pour le système Spark.
  2. Mémoire utilisateur :
    cette section est utilisée pour stocker toutes les structures de données définies (par exemple, les UDF de l'utilisateur) qui sont nécessaires aux opérations de conversion RDD. La mémoire utilisateur est gérée par Spark.
  3. Mémoire de stockage (unifiée) :
    Cette section est utilisée pour la mise en cache et la diffusion des données des tâches de traitement.
  4. Mémoire d'exécution (unifiée) :
    cette mémoire unifiée est utilisée pour les mélanges, les jointures, les tris et les agrégations.

Let’s do some simple calculations on 14 GBs (14,336 MBs) of the memory executor node and see the size of On-Heap Memory in each area.
Reserved Memory = 300 MBs
User Memory = (14,336 MBs — 300 MBs) * spark.memory.fraction
= (14,036 MBs) * 0.6 = 8,421 MBs = 8.2 GBs
Storage Memory = (14,336 MBs — 300 MBs) * spark.memory.fraction * (1- spark.memory.storageFraction) = 14,036 MBs* 0.6*0.5
= 4,210 MBs = 4.1 GBs
Execution Memory = (14,336 MBs — 300 MBs) * spark.memory.fraction * (1- spark.memory.storageFraction) = 14,036 MBs* 0.6*0.5
= 4,210 MBs = 4.1 GBs

Default values of Spark configuration are:

spark.memory.fraction = 0.6
spark.memory.storageFraction = 0.5

From all sections of memory in On-Heap Memory Structure,
Storage Memory and Execution Memory is unified memory.
It means these 2 sections have shared memory and can borrow from each other. This Spark internal process is called “Dynamic Occupancy Mechanism”.

Whenever total data size exceeds usable memory size (sum up from both execution task and storage task), this is when the spill occurs!

Off-Heap Memory by default is not enabled. If you want to use it,
you can set it up by using the following script.
spark.conf.set(spark.memory.offHeap.enabled, true)

Off-heap is like complementary memory in the worker node, but it is managed by the operating system so it does not have the overhead of garbage collection scanning. However, whenever Spark reads and writes data from off-heap into on-heap, it also adds the overhead on deserializing data from bytes arrays into Java bytes.

Possible Action to Generate Spill

Any action in a Spark job can lead to an increase in data size going over memory capacity.

  • Reducing spark.sql.shuffle.partitions leads to bigger file sizes per partition.
// Create a large partition by mismanaging shuffle partitions
spark.conf.set(“spark.sql.shuffle.partitions”, 60)
// The location of our non-skewed set of transactions
val trxPath = “./transactions/2011-to-2018_int-only_60gb.delta”spark
 .read.format(“delta”).load(trxPath) // Load the transactions table
 .orderBy(“trx_id”) // Some wide transformation
 .write.format(“noop”).mode(“overwrite”).save() // Execute a noop
  • join() or crossJoin() between 2 data tables.
  • Setting spark.sql.files.maxPartitionBytes to high (default is 128 MBs).
  • Do explode() on multiple column in data tables.

Atténuation des déversements

Comme vous l'avez peut-être réalisé, pour réduire le risque de déversement dans Spark, vous devez gérer l'espace mémoire au travail pour qu'il s'adapte bien à votre tâche. Vous trouverez ci-dessous la liste des choses à utiliser lorsque vous faites face à un problème de déversement.

  • Si le déversement est dû à un problème d’inclinaison, résolvez d’abord l’inclinaison !
  • Définissez le cluster avec plus de mémoire par travailleur (augmentez la taille du travailleur).
  • Gérez spark.sql.shuffle.partitions pour réduire la taille des fichiers par partition.
  • Effectuer . repartitioning() dans le code de transformation explicitement.
  • Définissez spark.sql.files.maxPartitionBytes pour qu'il corresponde à la nature de vos données.
